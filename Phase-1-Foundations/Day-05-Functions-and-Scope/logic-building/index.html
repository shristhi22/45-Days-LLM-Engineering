Logic Building · 30 minutes

How to Build
Logic When You Code

Syntax you can Google. The real skill is turning a problem into a plan a computer can follow. That skill is learnable — here's the method.

Think first. Type second.
The real bottleneck

You know the syntax. You're stuck anyway.

Most beginners can read every line of Python and still stare at a blank screen when handed a new problem. That's normal — and it's not a syntax gap.

It's a logic gap: the leap from "what is being asked" to "what steps solve it." Today is a repeatable method to cross that gap every time.

Example

Asked to "print a triangle of stars," you already know print() and for — but you freeze on how many stars each row needs. That missing step is the logic gap, not the syntax.

Definition

Logic building = designing the steps

🧠 The thinking job

Break a problem into a precise, ordered sequence of steps — clear enough that a machine (with zero common sense) can follow it.Task: greet 5 friends. Steps: repeat 5 times, each time print a name.

⌨️ The typing job

Translate those steps into Python. This is the easy part once the steps exist.for name in friends: print("Hi", name)

A computer does exactly what you say, in order, literally. Logic building is writing instructions for the world's fastest, most literal-minded helper.

The method

5 steps — and 4 are away from the keyboard

  • 1 · Understand — pin down the inputs, the output, and the rules.
  • 2 · Example by hand — solve one case on paper; watch how your brain does it.
  • 3 · Plan — write the steps in plain English (pseudocode); pick a pattern.
  • 4 · Code — translate each line of the plan into Python.
  • 5 · Test — run normal, boundary, and weird inputs; fix and repeat.
Example — "sum 1 to N" through all 5 steps

① in: N, out: total   ② by hand: 1+2+3 = 6   ③ total=0, loop & add   ④ for n in range(1,N+1): total += n   ⑤ test N=3→6, N=1→1, N=0→0

If you jump straight to step 4, you're guessing. The plan is what you're really paid for.

Step 1 of 5

Understand the problem

Don't write a line until you can answer three questions out loud:

📥 Input

What do I get, and in what form? A number? A string? A list?

📤 Output

What exactly must I produce or return? One value? Many?

📏 Rules

What are the conditions and the tricky edges (empty, zero, negative)?

Example — "10% off on bills of ₹500 or more"

Input: the bill amount (a number).   Output: the amount to pay.   Rule: discount only when bill ≥ 500; edge case = exactly 500.

Then restate it in one sentence of your own. Can't? You don't understand it yet — no code will fix that.

Step 2 of 5

Do one example by hand

Take a real input and solve it on paper — no code. Then ask the magic question:

The magic question

"What did my brain just do, step by step, to get that answer?"

Example — largest of 7, 3, 9, 4 (by hand)

keep the biggest seen so far: 7 → still 7 → now 9 → still 9. Answer: 9.
Your brain just ran a loop with a "best so far" variable — that is the algorithm.

Slow down and notice the steps you did automatically. Do a second example if the first felt like a fluke.

Step 3 of 5

Plan in plain English (pseudocode)

Write the steps, not the syntax
get the number N total = 0 for each number from 1 to N: add it to total show total
Why bother
  • Catches flawed logic before you fight the syntax.
  • Language-independent — same plan works in any language.
  • The comments for your code are already written.
Step 3, continued

The pattern toolbox

Most problems are a few familiar shapes combined. Learn to spot them:

➕ Accumulator

A running total or string you add to in a loop.total = 0 for p in prices: total += p

🔢 Counter

Tally how many times something happens.n = 0 for ch in text: if ch == "a": n += 1

🚩 Boolean flag

Track yes/no across a loop.failed = False for m in marks: if m < 40: failed = True

🔎 Search + break

Scan until you find it, then stop early.for x in items: if x == target: break

🏆 Best so far

Keep the current champion (max/min).best = 0 for m in marks: if m > best: best = m

🧩 Decompose

Split into small functions, solve each alone.avg = average(a, b, c) g = grade(avg)

Step 4 of 5

Translate the plan to code

Plan
total = 0 for each n from 1 to N: add n to total return total
Python
def add_up(N): total = 0 for n in range(1, N+1): total += n return total

Each plan line becomes one or two lines of code. Now the syntax from Days 1–5 is all you need: variables, loops, conditionals, functions.

Step 5 of 5

Test like you're trying to break it

Code that runs once on one input isn't done. Always try three kinds of input:

  • Normal — a typical case you expect (e.g. N = 5).
  • Boundary — the edges: 0, 1, empty string, the first/last item.
  • Weird — negatives, huge values, the unexpected.
Example — testing the bill function to_pay(bill)

to_pay(600) → 540 ✓ (normal)  ·  to_pay(500) → 450 ✓ (boundary)  ·  to_pay(0) → 0 ✓ (weird)

A failing test isn't failure — it's a free bug report. Trace it back to the plan, fix the logic, run again.

Worked example 1 · conditionals

Café bill — 10% off above ₹500

By hand: bill ₹600 is over ₹500 → take off ₹60 → pay ₹540.

Plan (step 3)
if bill >= 500: discount = bill * 0.10 else: discount = 0 final = bill - discount
Python (step 4)
def to_pay(bill): if bill >= 500: bill -= bill * 0.10 return bill print(to_pay(600)) # 540.0

Test (step 5): 600 → 540 · 500 → 450 (boundary) · 499 → 499.

Worked example 2 · loop + best-so-far

Find the class topper

By hand: marks 72, 88, 65 → keep the biggest seen so far: 72 → 88 → 88. Topper = 88.

Plan (step 3)
highest = 0 for each of the 5 marks: read mark if mark > highest: highest = mark show highest
Python (step 4)
highest = 0 for i in range(5): mark = int(input("Mark: ")) if mark > highest: highest = mark print("Topper:", highest)

Pattern: "best so far" — one champion variable, updated only when beaten.

Worked example 3 · loop + counter

Count the vowels in a name

By hand: "Saanvi" → S a a n v i → vowels a, a, i → 3.

Plan (step 3)
count = 0 for each letter in the name: if letter is a vowel: count = count + 1 return count
Python (step 4)
def count_vowels(name): count = 0 for ch in name.lower(): if ch in "aeiou": count += 1 return count print(count_vowels("Saanvi")) # 3

Pattern: "counter" — start at 0, add 1 each time a condition is true.

Worked example 4 · decompose into functions

Marks → grade (two small functions)

Big task, split in two: first the average, then the grade band. Solve each alone.

Plan (step 3)
average = (a + b + c) / 3 if average >= 90: grade "A" elif average >= 75: grade "B" elif average >= 40: grade "C" else: grade "F"
Python (step 4)
def average(a, b, c): return (a + b + c) / 3 def grade(avg): if avg >= 90: return "A" if avg >= 75: return "B" if avg >= 40: return "C" return "F" print(grade(average(82, 91, 70))) # B

Pattern: "decompose" — two simple functions beat one tangled block.

Train your eye

Real task → which pattern?

Once you know the toolbox, you start seeing the pattern inside the words:

🛒 Cart total

Add up every item's price → accumulator.

💬 Long messages

How many texts are over 160 chars → counter.

❌ Any backlog?

Did any subject fail → flag / search + break.

🏆 Class topper

Highest mark in the room → best so far.

🔢 Times table

Print 7×1 … 7×10 → loop + range.

📱 Valid mobile?

Exactly 10 digits → conditionals + string check.

When it breaks

Stuck? Debug the logic, not the keyboard

🖨️ Print to see

Drop in print() to watch variables change. Reality vs. what you assumed — the gap is the bug.

🦆 Explain it aloud

Talk through your code line by line (to a friend, or a rubber duck). You'll hear the flaw yourself.

✂️ Shrink the input

Make it the smallest failing case. Less noise, faster fix.

↩️ Back to the plan

Most bugs are wrong logic, not wrong syntax. Re-check step 3, not the semicolons.

Example — print-debugging a wrong total

Sum of 1..5 prints 10, not 15? Add print(n, total) in the loop — you see it stops at 4. The bug: range(1, N) should be range(1, N+1).

Avoid these

Beginner traps

  • Coding before understanding — typing to "figure it out" wastes the most time.
  • Trying to write it perfectly in one go — get it working, then improve it.
  • Copying code you don't understand — it breaks the moment the problem shifts.
  • Ignoring edge cases — empty, zero, and negative inputs are where bugs hide.
  • Do instead: plan on paper, build in small steps, test as you go.
Example — the edge case that bites

An "average of a list" function works on [5, 8] but crashes on an empty list (divide by zero). Thinking about edges before coding catches it.

It's a muscle

How logic building actually grows

Nobody is "born logical." It's pattern recognition built by reps — the more problems you solve, the faster you see the shape of a new one.

📅 A little, daily

One or two problems a day beats a weekend binge. Consistency wins.

✍️ Plan first, always

Force the pseudocode step even when you "know" the answer.

🔁 Re-solve

Redo old problems a different way. New angles deepen the patterns.
